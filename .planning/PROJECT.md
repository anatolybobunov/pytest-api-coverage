# pytest-api-coverage — Multi-Spec Feature

## What This Is

pytest-api-coverage — это pytest-плагин, который перехватывает HTTP-запросы во время тестов и сопоставляет их с Swagger/OpenAPI спецификациями, формируя отчёты о покрытии API. Сейчас плагин поддерживает одну спецификацию на запуск. Цель этой работы — добавить поддержку нескольких спецификаций одновременно, маппируя URL-сервисов на конкретную спеку и генерируя отдельный отчёт на каждую.

## Core Value

Разработчики могут запустить один `pytest` и получить отдельные отчёты покрытия для каждого микросервиса, не меняя тестовый код.

## Requirements

### Validated

- ✓ Перехват HTTP-запросов через `requests` и `httpx` — existing
- ✓ Парсинг Swagger 2.0 и OpenAPI 3.x из файлов и URL — existing
- ✓ Сопоставление запросов с эндпоинтами по пути и методу — existing
- ✓ Генерация отчётов в форматах JSON, CSV, HTML — existing
- ✓ Поддержка параллельного выполнения через pytest-xdist — existing
- ✓ Фильтрация запросов по origin (base_url / include_base_urls) — existing
- ✓ Опция `--swagger` для одной спецификации — existing

### Active

- [ ] Поддержка нескольких спецификаций через конфиг-файл (YAML/JSON)
- [ ] Автодетект `coverage-config.yaml` / `coverage-config.json` в корне проекта
- [ ] CLI-флаги `--coverage-spec-name`, `--coverage-spec-path`, `--coverage-spec-url` для одной спеки
- [ ] Именование выходных файлов с префиксом (`auth-coverage.html`)
- [ ] Terminal summary с одной строкой на спеку и счётчиком проигнорированных запросов
- [ ] Обратная совместимость: `--swagger` продолжает работать без изменений
- [ ] xdist совместимость для multi-spec режима
- [ ] Валидация: запрет совмещать `--swagger` с multi-spec флагами

### Out of Scope

- Слияние отчётов нескольких спек в один — каждая спека получает свой независимый отчёт
- Динамическое переключение спек во время одного тестового сеанса — все спеки фиксируются на старте
- Поддержка других форматов конфига кроме YAML и JSON

## Context

Плагин построен на layered архитектуре: adapters (перехват) → collector (хранение) → reporter (анализ) → writers (вывод). `CoverageReporter` уже поддерживает фильтрацию по URL через `include_base_urls` — именно этот механизм используется для маппинга запросов на спеки. Новый `MultiSpecOrchestrator` оркестрирует несколько экземпляров `CoverageReporter`, каждый со своим набором URL.

Технический стек: Python 3.11+, pytest 7+, PyYAML (уже в зависимостях), mypy strict mode, ruff.

## Constraints

- **Обратная совместимость**: `--swagger` должен работать идентично текущему поведению — нельзя ломать существующих пользователей
- **Fail-silent принцип**: ошибки в multi-spec конфигурации не должны ломать тесты (предупреждения, не исключения)
- **xdist**: workers только собирают данные, не читают спеки — спеки загружаются только на мастере
- **Зависимости**: PyYAML уже есть; новых runtime-зависимостей добавлять не нужно

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| `MultiSpecOrchestrator` поверх существующих `CoverageReporter` | Максимальное переиспользование, минимальный риск регрессий | — Pending |
| Конфиг-файл как основной способ для multi-spec | CLI плохо масштабируется для структурированных данных с повторяющимися группами | — Pending |
| Игнорировать запросы вне всех URL + счётчик в terminal | Не ломает fail-silent; пользователь видит что запросы не потеряны | — Pending |
| Swagger-файлы читаются только на мастере xdist | Workers не нуждаются в спеках; упрощает сериализацию | — Pending |
